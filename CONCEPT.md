# LogiFlow — Концепція продукту

## Суть продукту

LogiFlow — SaaS-платформа для автоматизації планування логістичних доставок по дворівневому ланцюжку:

**Завод → Склад → Магазин**

Компанія підписує контракт з клієнтом (напр. Аврора), клієнт надає дані своєї мережі (склади, магазини, заводи-партнери, вантажівки), система будує оптимальні маршрути доставки з урахуванням пріоритетності точок і вартості перевезення.

---

## Ключова ідея

Класична задача маршрутизації (VRP) + **пріоритизація за критичністю попиту**.

Система не просто мінімізує кілометри — вона враховує **наскільки терміново** потрібен товар в кожній точці і будує маршрут так, щоб критичні магазини отримали поставку першими.

### Unfair advantage

> Єдина система що з'єднує планування двох рівнів доставки в єдиний потік: вантажівка на шляху завод→склад автоматично отримує пропозицію попутної доставки в критичні точки — з розрахунком фінансової вигоди, дотриманням законодавства про режим водія (EC 561/2006) і повним timeline рейсу. SAP TM, Google OR-Tools і аналоги планують кожен рівень окремо і не дають цього зв'язку.

## Вхідні дані

Клієнт завантажує 4 CSV-файли:

| Файл | Що містить |
|---|---|
| `nodes.csv` | Всі вузли мережі: заводи, склади, магазини (координати, тип, ємність) |
| `edges.csv` | Граф відстаней між вузлами (пари + км) |
| `trucks.csv` | Вантажівки: місткість, вартість на км, поточне місце |
| `demand.csv` | Поточний попит: скільки товару потрібно кожному магазину і наскільки терміново |

Альтернативно — система генерує тестовий датасет рандомно (довільна кількість вузлів).

---

## Ланцюжок доставки

```
FACTORY ──[Leg 1]──► WAREHOUSE ──[Leg 2]──► STORE
```

- **Leg 1** — вантажівки везуть товар з заводів на склади
- **Leg 2** — вантажівки везуть товар зі складів у магазини
- Жодна вантажівка не їде Завод→Магазин напряму (бізнес-правило)
- Кожен Leg — окреме планування маршрутів

---

## Пріоритизація

Кожному магазину присвоюється рівень критичності на основі залишку товару:

| Рівень | Умова | Ваговий коефіцієнт |
|---|---|---|
| NORMAL | запас > 50% мінімуму | 1.0 |
| ELEVATED | запас 20–50% мінімуму | 2.0 |
| CRITICAL | запас < 20% або urgent-запит | 4.0 |

CRITICAL-точки мають пріоритет при розподілі вантажівок і при побудові маршруту.

---

## Алгоритмічне ядро

1. **Dijkstra** (з heapq) — знаходить найкоротші шляхи між всіма вузлами → матриця відстаней
2. **Greedy assignment з пріоритетами** — розподіляє замовлення між вантажівками, мінімізуючи `вартість_на_км × відстань / пріоритет`
3. **TSP nearest-neighbor** — будує маршрут для кожної вантажівки (next = найближча нерозвезена точка з урахуванням пріоритету)
4. **2-Opt** — локальна оптимізація маршруту (скорочення на 20-40%)

---

## Гнучкість (ключова вимога)

Система не прив'язана до фіксованої топології:
- Довільна кількість заводів, складів, магазинів
- Довільна кількість вантажівок з різними параметрами
- Для тестування — рандомна генерація мережі за параметрами (seed для відтворюваності)

```
Generate: 3 заводи, 7 складів, 40 магазинів, 12 вантажівок → система вирішує
```

## Склади та запаси

### Склад — не просто вузол

Кожен склад має власний інвентар: що є і скільки. Це ключова інформація для алгоритму — без неї неможливо знати звідки брати товар для конкретного магазину.

```
WAREHOUSE_1:
  Місткість: 10,000 кг
  Зайнято:    6,400 кг
  Вільно:     3,600 кг

  Інвентар:
    product_A: 2,000 кг
    product_B:   800 кг
    product_C: 3,600 кг
```

### Новий файл — warehouse_stock.csv

```csv
warehouse_id, product_id, quantity_kg
WAREHOUSE_1,  product_A,  2000
WAREHOUSE_1,  product_B,  800
WAREHOUSE_2,  product_A,  50
WAREHOUSE_2,  product_C,  1200
```

### Як це впливає на алгоритм assignment

Коли магазин просить товар — система шукає не просто найближчий склад, а найближчий склад **де є потрібний товар у достатній кількості**:

```
STORE_7 потребує: 85 кг product_A, пріоритет CRITICAL

WAREHOUSE_1: product_A = 2,000 кг ✅  відстань 45 км  → кандидат
WAREHOUSE_2: product_A = 50 кг    ❌  замало           → пропускаємо
WAREHOUSE_3: product_A = 500 кг   ✅  відстань 120 км  → кандидат

→ Обираємо WAREHOUSE_1 (ближче і є товар)
```

### Що якщо жоден склад не має достатньо

Три варіанти в порядку пріоритету:

1. **Розділити між складами** — взяти частину з WAREHOUSE_1, решту з WAREHOUSE_3
2. **Сигнал на Leg1** — склади порожні, потрібне поповнення з заводу спочатку
3. **Попередження диспетчеру** — "замовлення STORE_7 не може бути закрите наявними запасами"

### Оновлення запасів після доставки

Після підтвердження маршруту система автоматично **резервує** товар на складі:

```
WAREHOUSE_1 product_A: 2,000 кг → 1,915 кг (зарезервовано 85 кг для STORE_7)
```

Після фактичної доставки — запас списується остаточно.

## Urgent Request

Магазин або диспетчер може надіслати термінований запит:
- Точка автоматично стає CRITICAL
- Система перераховує маршрути для affected вантажівок
- Диспетчер бачить diff: що змінилось

## Попутна доставка (En-Route Delivery)

### Ідея

Під час побудови маршруту Leg2 (Склад→Магазини) система перевіряє: чи є **CRITICAL магазини** поблизу вже запланованого маршруту між зупинками? Якщо так — пропонує диспетчеру додати зупинку по дорозі без окремого рейсу.

Диспетчер бачить пропозицію з розрахунком і сам приймає рішення.

### Чому тільки Leg2, не Leg1

Leg1 — це фури і великі вантажівки (завод→склад). Вони:
- не можуть в'їхати в місто (обмеження тоннажу)
- не можуть запаркуватись біля міського магазину
- не розкривають опломбований причіп заради малої партії

Leg2 — малі вантажівки і буси (3-5 т) вже їдуть зі складу по магазинах у місті. Вони і так роблять зупинки — логічно перевірити чи є CRITICAL точка по дорозі.

```
НЕПРАВИЛЬНО (фура не може):
Semi: Завод → [заїзд в Аврору] → Склад

ПРАВИЛЬНО (бус у місті):
Van: Склад → Аврора_A → [Аврора_B по дорозі, CRITICAL] → Аврора_C
```

### Типи транспорту в trucks.csv

```csv
id,  type,   capacity_kg,  fuel_per_100km,  depot_node_id
T1,  semi,   20000,        35,              FACTORY_1
T2,  truck,  5000,         28,              WAREHOUSE_1
T3,  van,    1500,         12,              WAREHOUSE_2
```

**Правило:** En-Route пропонується тільки для `truck` і `van`. Тип `semi` — тільки прямий маршрут Leg1, без заїздів у магазини.

---

### Концепція алгоритму

#### Крок 1 — Побудувати базовий маршрут Leg2
```
Route = [WAREHOUSE → STORE_A → STORE_C → STORE_E → WAREHOUSE]
```

#### Крок 2 — Знайти кандидатів між зупинками
Для кожного CRITICAL магазину що не в маршруті — перевіряємо відхилення між кожною парою сусідніх зупинок:

```
detour_km = dist(STORE_A → STORE_B_candidate) 
            + dist(STORE_B_candidate → STORE_C)
            - dist(STORE_A → STORE_C)

detour_ratio = detour_km / total_route_km
```

Магазин є кандидатом якщо:
- `detour_ratio < threshold` (15% — налаштовується)
- Вантажівка має місце для товару магазину (chargeable_weight)
- Магазин має статус CRITICAL або ELEVATED

#### Крок 3 — Розрахувати економію vs витрати

```
cost_with_stop    = (базова_відстань + detour_km) × cost_per_km
cost_without_stop = базова_відстань × cost_per_km 
                  + окремий_рейс_до_магазину × cost_per_km

savings = cost_without_stop - cost_with_stop
```

Якщо `savings > 0` — попутна доставка вигідна фінансово.
Якщо `savings < 0` але магазин CRITICAL — система все одно пропонує, але з позначкою "невигідно фінансово, виправдано пріоритетом".

#### Крок 4 — Вставити зупинку в маршрут
```
Route = [WAREHOUSE → STORE_A → STORE_B (нова) → STORE_C → WAREHOUSE]
```

### Параметри (налаштовуються клієнтом)

| Параметр | За замовчуванням | Опис |
|---|---|---|
| `max_detour_ratio` | 15% | Максимальне відхилення від маршруту |
| `min_priority` | ELEVATED | Мінімальний пріоритет для пропозиції |
| `eligible_types` | truck, van | Типи авто для En-Route (semi — виключено) |
| `show_unprofitable` | true | Показувати навіть невигідні якщо CRITICAL |

---

### Чому це інноваційно

Стандартні системи будують маршрут Leg2 один раз і не переглядають його в контексті незакритих CRITICAL точок. Ця фіча аналізує геометрію вже побудованого маршруту і знаходить "дірки" — точки які можна закрити майже безкоштовно. Диспетчер отримує підказку яку раніше міг дати тільки досвідчений логіст знаючи мережу напам'ять.

## Робота в умовах нестабільного зв'язку

### Підхід: проактивна синхронізація, не реактивний offline

Система не "реагує на відсутність зв'язку" — вона **проактивно доставляє все потрібне до моменту втрати зв'язку**. Водій виїжджає з депо де завжди є WiFi. Все що йому потрібно на день — вже відомо до виїзду і збережено локально.

```
Диспетчер натискає Solve
  → маршрути розраховані на backend
  → кожен маршрут пушиться відповідному водієві (або водій відкриває app)
  → дані зберігаються в IndexedDB на телефоні водія
  → водій їде, зв'язку немає → дані вже є локально → все працює
```

---

### Що працює offline (read-only + черга)

**Перегляд маршруту:**
- Список зупинок на сьогодні в правильному порядку
- Для кожної зупинки: назва точки, адреса, товар, кількість кг, пріоритет
- Timeline з часами прибуття і перервами (вже порахований на backend)

**Дії в черзі (queue):**
Якщо водій виконує дію без інтернету — вона зберігається локально і відправляється автоматично коли з'явиться зв'язок:

| Дія | Поведінка offline |
|---|---|
| Надіслати Urgent Request | → черга → відправка при підключенні |
| Підтвердити доставку на точці | → черга → відправка при підключенні |
| Позначити перерву | → черга → відправка при підключенні |

Водій бачить індикатор: `📡 Офлайн — 2 дії в черзі`

**Недоступно offline:**
- Solve нового маршруту (алгоритм на backend)
- Завантаження нових CSV
- Реалтайм оновлення від диспетчера

---

### Конфлікти при поверненні в мережу

**Сценарій:** водій був offline, диспетчер змінив його маршрут через urgent request іншого магазину.

**Правило:** дані сервера завжди мають пріоритет над локальними.

**Алгоритм merge при reconnect:**
```
1. Відправити локальну чергу дій на сервер
2. Завантажити актуальний маршрут з сервера
3. Порівняти з локальною версією
4. Якщо маршрут змінився → показати банер:
   "⚠️ Маршрут оновлено диспетчером. [Переглянути зміни]"
5. Замінити локальну версію серверною
```

### Технічна реалізація

**PWA (Progressive Web App):**
```
vite-plugin-pwa → автоматично генерує Service Worker
Service Worker кешує:
  - app shell (index.html, JS, CSS) → завжди завантажується
  - /api/routes/my → маршрут поточного водія
```

**IndexedDB (локальне сховище):**
```javascript
// Структура сховища:
stores:
  - current_route      // маршрут водія на сьогодні
  - action_queue       // черга дій для відправки
  - last_sync_at       // timestamp останньої синхронізації
```

**Background Sync API:**
```javascript
// При появі зв'язку:
self.addEventListener('sync', event => {
  if (event.tag === 'action-queue') {
    event.waitUntil(flushActionQueue())
  }
})
```

**Індикатор стану в UI:**
```
Online:  🟢 Підключено | Оновлено: 14:32
Offline: 🔴 Офлайн | 2 дії в черзі | Останнє оновлення: 14:32
```

### Що треба зробити в коді (чекліст для команди)

**Frontend:**
- [ ] Встановити і налаштувати `vite-plugin-pwa`
- [ ] Визначити які routes кешує Service Worker (`/api/routes/my`, static assets)
- [ ] Реалізувати `useOfflineQueue` hook — додає дії в IndexedDB якщо offline
- [ ] Реалізувати `useSyncStatus` hook — показує індикатор стану і кількість дій в черзі
- [ ] При reconnect — запустити merge: flush queue → fetch свіжий маршрут → показати diff якщо змінився

**Backend:**
- [ ] `GET /api/routes/my` — маршрут поточного водія (кешується SW)
- [ ] `POST /api/actions/batch` — прийняти пачку дій з черги (з timestamp кожної)
- [ ] Логіка merge: якщо дія в черзі конфліктує з поточним станом → сервер виграє, повернути актуальний стан

## Інтерфейс (MVP)

### Один додаток — дві вкладки

Для MVP: один веб-додаток з перемикачем ролі в хедері. Без логіну і JWT — суддям важливо побачити алгоритм, не авторизацію.

```
┌─────────────────────────────────────────┐
│ LogiFlow    [🗺 Диспетчер] [📦 Склад]   │
└─────────────────────────────────────────┘
```

---

### Вкладка 1 — Диспетчер (Desktop)

Головний екран з картою і керуванням.

**Що є:**
- Інтерактивна карта: вузли з кольорами за пріоритетом 🔴🟡🟢, маршрути вантажівок лініями
- Upload CSV або кнопка Generate (задати кількість вузлів)
- Кнопка **Solve** → маршрути будуються і відображаються на карті
- Панель En-Route пропозицій після Solve
- Таблиця маршрутів: вантажівка, зупинки, км, вартість грн, timeline
- Кнопка **Urgent** на будь-якій точці → re-solve → показати diff

---

### Вкладка 2 — Склад (Mobile-friendly)

Спрощений мобільний вигляд для оператора складу.

**UX:**
- Великі контрастні кнопки (зручно в рукавицях)
- Темна тема
- Мінімум тексту

**Що є:**
- Список замовлень на відвантаження сьогодні
- Для кожного: товар, кількість, куди, яка вантажівка
- Кнопка "Відвантажено" → статус оновлюється
- Urgent у 2 тапи: вибрати товар → підтвердити
- Індикатор offline якщо немає зв'язку

```
┌─────────────────────────────┐
│ 📦 Замовлення #47           │
│ product_A  →  85 кг         │
│ Для: Аврора Позняки 🔴      │
│ Вантажівка: T2              │
│                             │
│  [✅ ВІДВАНТАЖЕНО]          │
└─────────────────────────────┘
```

- CRITICAL магазини ніколи не залишаються без поставки через "не встигли"
- Вартість маршруту прозора: видно скільки коштує кожен рейс
- Зміна попиту → система перераховує за секунди, не за годину

## Планування часу рейсу (MVP)

Після побудови маршруту система будує **повний timeline** з часом відправлення, прибуття до кожної зупинки і обов'язковими перервами. Диспетчер і водій бачать не "~6 год", а реальний розклад.

### Вхідний параметр

Диспетчер задає **час виїзду** перед натисканням Solve (або лишає дефолт `08:00`). Всі часи рахуються відносно нього.

### Що будується для кожного маршруту

```
🚐 T2 — Вантажівка-1  |  87 км  |  2,248 грн
─────────────────────────────────────────────
08:00  ▶ Виїзд          Склад Центр
08:22  📍 Прибуття      Аврора Позняки      🔴 CRITICAL  85 кг
08:37  ▶ Виїзд          Аврора Позняки      (15 хв розвантаження)
09:00  📍 Прибуття      Аврора Лівобережна  🟡 ELEVATED  40 кг
09:15  ▶ Виїзд          Аврора Лівобережна
09:45  🏠 Повернення    Склад Центр

Загальний час рейсу: 1 год 45 хв  (їзда: 1 год 35 хв + розвантаження: 30 хв)
```

### Алгоритм побудови timeline (backend)

```python
BREAK_AFTER_H   = 4.5   # EC 561/2006: перерва після 4.5 год їзди
BREAK_MIN       = 45    # хвилин перерви
UNLOAD_MIN      = 15    # хвилин на розвантаження на кожній зупинці (дефолт)
DAILY_LIMIT_H   = 9.0   # максимум їзди за день

def build_timeline(stops, dist_matrix, departure_time, avg_speed, unload_min=15):
    current_time = departure_time      # datetime об'єкт
    driven_since_break = 0.0           # год за кермом без перерви
    timeline = []

    timeline.append({
        "node_id": stops[0],
        "time": fmt(current_time),
        "event": "departure",
        "note": None
    })

    for i in range(len(stops) - 1):
        seg_km = dist_matrix[stops[i]][stops[i+1]]
        seg_h  = seg_km / avg_speed

        # Якщо цей відрізок перетинає поріг 4.5 год — вставити перерву
        if driven_since_break + seg_h > BREAK_AFTER_H:
            current_time += timedelta(minutes=BREAK_MIN)
            driven_since_break = 0.0
            timeline.append({
                "node_id": None,
                "time": fmt(current_time),
                "event": "break",
                "note": "Обов'язкова перерва 45 хв (EC 561/2006)"
            })

        current_time += timedelta(hours=seg_h)
        driven_since_break += seg_h
        is_last = (i == len(stops) - 2)

        timeline.append({
            "node_id": stops[i+1],
            "time": fmt(current_time),
            "event": "return" if is_last else "arrival",
            "note": None
        })

        # Час на розвантаження (крім останньої зупинки — повернення на склад)
        if not is_last:
            current_time += timedelta(minutes=unload_min)
            timeline.append({
                "node_id": stops[i+1],
                "time": fmt(current_time),
                "event": "departure",
                "note": f"{unload_min} хв розвантаження"
            })

    total_drive_h = sum(dist_matrix[stops[i]][stops[i+1]]
                        for i in range(len(stops)-1)) / avg_speed

    # Визначити тип рейсу
    if total_drive_h <= BREAK_AFTER_H:
        days, status, warning = 1, "ok", None
    elif total_drive_h <= DAILY_LIMIT_H:
        days, status = 1, "warning"
        warning = "Потрібна перерва 45 хв (вже включена в timeline)"
    elif total_drive_h <= DAILY_LIMIT_H * 2:
        days, status = 2, "multiday"
        warning = "Дводобовий рейс — водій потребує ночівлі між днями"
    else:
        days, status = 3, "multiday"
        warning = "Рейс 3+ днів — потребує окремого планування диспетчером"

    return {
        "timeline": timeline,
        "departure_time": fmt(departure_time),
        "arrival_time": fmt(current_time),
        "total_drive_h": round(total_drive_h, 1),
        "total_elapsed_h": round((current_time - departure_time).seconds / 3600, 1),
        "days": days,
        "status": status,   # "ok" | "warning" | "multiday"
        "warning": warning
    }
```

### Що показується в таблиці маршрутів (summary рядок)

```
Вантажівка | Маршрут               | Км    | Виїзд | Повернення | Днів | Вартість
───────────┼───────────────────────┼───────┼───────┼────────────┼──────┼──────────
T2         | Склад → 3 магазини    | 87 км │ 08:00 │ 09:45  ✅  │  1   │ 2,248 грн
T1         | Завод → Склад         | 520 км│ 08:00 │ 18:12  ⚠️  │  1   │ 12,400 грн
T4         | Завод → Склад (далек.)| 780 км│ 08:00 │ +1 день 🔴 │  2   │ 18,600 грн
```

Клік на рядок → розгортається повний timeline зупинок з часами.

### Що повертає `/api/solve` і `/api/routes/{truck_id}`

Кожен маршрут містить:
- `departure_time` — час виїзду
- `arrival_time` — час повернення на депо
- `total_drive_h` — чистий час за кермом
- `total_elapsed_h` — загальний час рейсу (їзда + розвантаження + перерви)
- `days` — 1, 2 або 3
- `status` / `warning` — для підсвічування в таблиці
- `timeline[]` — масив подій з часами (виїзд, прибуття, перерва, повернення)

## Об'ємна вага (Dimensional Weight)

### Ідея

Вантажівка "повна" не тільки коли вичерпана вага — а й коли закінчується місце. Щоб не рахувати окремо вагу і об'єм, використовуємо **один параметр** — об'ємну вагу. Це стандарт автоперевезень (DHL, Нова Пошта, Meest).

### Формула

```
volumetric_weight_kg = (length_cm × width_cm × height_cm) / 4000

chargeable_weight_kg = max(actual_weight_kg, volumetric_weight_kg)
```

Дільник `4000` — галузевий стандарт для наземних перевезень.
Для авіа — 6000, для кур'єрів — 5000.

**Приклад:**
```
Товар: 8 кг, розміри 60×40×50 см
volumetric = (60×40×50) / 4000 = 30 кг
chargeable = max(8, 30) = 30 кг  ← береться більше
```

### Як використовується в системі

Вантажівка "повна" коли:
```
sum(chargeable_weight) для всіх вантажів >= truck.capacity_kg
```

Один параметр замість двох окремих перевірок (вага + об'єм).

### Вхідні дані

**products.csv** (або в demand.csv):
```csv
product_id, weight_kg, length_cm, width_cm, height_cm
product_A,  8,         60,        40,        50
product_B,  25,        40,        30,        30
```

Розміри — **опціональні**. Якщо не вказані — система використовує тільки `weight_kg`. Клієнт заповнює що знає.

### Що це дає

- Алгоритм автоматично уникає ситуації "вантажівка легка але забита під стелю"
- Одна формула — один параметр в алгоритмі
- Реальна індустріальна практика → суддям не можна заперечити

## Формула вартості доставки

### Ідея

Клієнт вказує лише те що знає про свій флот. Решта — дефолтні значення. Одна глобальна змінна (ціна пального) оновлюється окремо і автоматично перераховує все.

### Глобальне налаштування (один раз)

```
Ціна пального: [_52.0_] грн/л
```

### Параметри вантажівки (trucks.csv)

Клієнт заповнює лише 3 обов'язкових поля:

```csv
id, capacity_kg, fuel_per_100km, depot_node_id
T1, 5000,        28,             WAREHOUSE_1
T2, 3000,        22,             WAREHOUSE_2
```

Решта підтягується з дефолтів (можна перевизначити):

```csv
driver_hourly      = 180 грн    ← середня ставка водія
avg_speed_kmh      = 55 км/год  ← середня по дорогах України
amortization_per_km = 5.0 грн  ← ~3M грн / 600K км
maintenance_per_km  = 3.0 грн  ← шини, масло, ТО
```

### Формула (рахує система)

```
cost_per_km = (fuel_per_100km / 100) × fuel_price
            + driver_hourly / avg_speed_kmh
            + amortization_per_km
            + maintenance_per_km
```

Приклад для T1:
```
= (28/100) × 52  →  14.56 грн  (паливо)
+ 180 / 55       →   3.27 грн  (водій)
+ 5.0            →   5.00 грн  (амортизація)
+ 3.0            →   3.00 грн  (обслуговування)
─────────────────────────────
= 25.83 грн/км
```

### UX: як це виглядає в інтерфейсі

Клієнт бачить просту картку для кожної машини:

```
┌─────────────────────────────────────┐
│ Вантажівка T1                       │
│ Витрата пального: [28] л/100км      │
│ Місткість: [5000] кг                │
│                                     │
│ ▼ Розширені налаштування            │
│   Ставка водія: [180] грн/год       │
│   Середня швидкість: [55] км/год    │
│   Амортизація: [5.0] грн/км         │
│   Обслуговування: [3.0] грн/км      │
│                                     │
│ Розрахована вартість: 25.83 грн/км  │
└─────────────────────────────────────┘
```

Розширені налаштування — приховані за замовчуванням. Більшість клієнтів заповнить лише 2 поля.

### Що дає ця формула системі

- При зміні ціни пального — всі маршрути автоматично перераховуються
- Різні вантажівки мають реальну різну вартість → алгоритм обирає дешевшу де можливо
- У звіті диспетчер бачить точну вартість кожного рейсу в гривнях

## Захищеність інформації (MVP)

Для демо — тільки те що реально потрібне і не займає час:

**Pydantic валідація CSV** — єдине що варто зробити. Захищає від некоректних даних і падінь алгоритму:
```python
class DemandRow(BaseModel):
    node_id: str
    current_stock: float = Field(ge=0)
```

**Перемикач ролі в хедері** — для демо достатньо. Судді бачать різні вкладки, не бачать JWT.

## Архітектурні рішення (для команди)

Три рішення які впливають на код і мають бути зафіксовані до початку реалізації:

### 1. Multi-tenant
**Рішення: ні для MVP.**
Один деплой — один клієнт (демо Аврора). Не ускладнюємо БД `client_id`. Якщо буде SaaS модель в майбутньому — в ROADMAP.

### 2. API як окремий інтерфейс
**Рішення: так, автоматично.**
Backend на FastAPI — solver доступний через API незалежно від UI. Solver є окремим модулем який викликається і через UI і напряму через API. Нічого додаткового робити не треба.

### 3. Розмір даних — кількість вузлів
**Рішення: універсальний, без жорсткого ліміту.**

Система не прив'язана до конкретного розміру мережі. Алгоритм працює однаково для 10 і для 10,000 вузлів. Обмеження лише в часі виконання:

| Розмір мережі | Вузлів | Час Solve |
|---|---|---|
| Малий (демо) | 10–50 | < 1 сек |
| Середній (реальний клієнт) | 50–500 | 1–10 сек |
| Великий (мережа типу АТБ) | 500–5000 | 10–60 сек |

Для великих мереж — Dijkstra з heapq справляється. Якщо треба швидше: кешування матриці відстаней між сесіями (не перераховувати якщо граф не змінився).

Генератор тестових даних підтримує будь-який розмір через параметри:
```
n_factories, n_warehouses, n_stores, n_trucks, seed
```
