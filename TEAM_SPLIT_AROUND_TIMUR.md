# Team Split Around Timur

Цей документ фіксує правильний поділ роботи на **двох людей навколо Тимура**, щоб команда рухалась паралельно і не перемішувала алгоритм, сервер і сайт.

Логіка поділу така:

- **Тимур** не рухається зі своєї зони
- навколо нього є ще **2 окремі ролі**
- разом це дає 3 незалежні, але пов'язані частини продукту:
  - **Тимур = solver / алгоритм**
  - **Людина 1 = Backend / API / Database / Integration**
  - **Людина 2 = Frontend / SPA / UI**

---

## 1. Головний принцип поділу

У концепті LogiFlow продукт виглядає так:

```text
дані мережі -> solver -> маршрути -> API -> сайт -> менеджер / склад
```

Тому і команда має бути поділена по цьому ж ланцюжку:

### Тимур
Відповідає за:
- математичну логіку
- shortest paths
- пріоритети
- assignment
- побудову маршрутів
- route metrics

### Backend / API людина
Відповідає за:
- зберігання і читання даних
- виклик solver-а Тимура
- перетворення результату в API
- збереження routes, cargo, stock changes

### Frontend / SPA людина
Відповідає за:
- сайт
- вкладки для ролей
- візуалізацію мережі
- показ попиту, маршрутів, завантаження
- дії користувача через UI

---

## 2. Тимура не чіпаємо

Це важливо.

Тимура **не треба змішувати** ні з frontend, ні з базою, ні з деплоєм, ні з дизайном.

Його зона:

```text
отримати структуровані дані -> порахувати -> повернути структурований результат
```

Його ключовий результат:

- solver для `Leg 2`
- shortest path matrix
- assignment замовлень по машинах
- маршрут кожної машини
- метрики:
  - `total_km`
  - `total_cost`
  - `drive_hours`
  - `total_elapsed_h`
- список stop-ів
- cargo по кожному маршруту

Тобто Тимур відповідає на питання:

```text
як саме система вирішила, хто куди їде і що везе
```

Все інше навколо цього вже збирають інші двоє.

## 3. Роль 1 — Backend / API / Database / Integration

### Суть ролі

Ця людина будує **серверну частину системи**.

Вона відповідає за те, щоб:

- дані були збережені
- solver мав звідки брати вхідні таблиці
- frontend мав звідки брати готові результати
- натискання кнопок у UI реально запускали потрібну логіку

Простою мовою:

```text
ця людина з'єднує базу, алгоритм Тимура і майбутній сайт в одну працюючу систему
```

### Чому ця роль критична

Без неї:

- solver Тимура існуватиме сам по собі
- frontend матиме лише красивий макет
- дані не будуть проходити через продукт

Саме ця людина робить так, щоб продукт був **не окремими модулями**, а **цілісним сервісом**.

### Що саме вона повинна зробити

#### 1. Підняти базу даних по концепту

Треба створити SQLite schema згідно [DATABASE.md](/Users/yurleomel/Desktop/BestHackaton/DATABASE.md).

Ключові таблиці:

- `nodes`
- `edges`
- `products`
- `trucks`
- `warehouse_stock`
- `demand`
- `routes`
- `route_cargo`
- `settings`

Це критично, бо саме ці таблиці є джерелом істини для всього продукту.

#### 2. Зробити завантаження demo_data

Треба імпортувати CSV з `demo_data/` у БД.

Це потрібно, щоб:

- команда могла одразу працювати з реальним тестовим набором
- фронтенд міг уже зараз читати живі дані
- solver Тимура міг брати нормальний input

#### 3. Реалізувати REST API

Потрібні базові endpoint-и:

- `GET /api/network`
- `GET /api/routes`
- `GET /api/demand`
- `GET /api/stock`
- `POST /api/solve`
- `POST /api/stock/update`

Можливо також:

- `POST /api/urgent`

#### 4. Підключити solver Тимура

Найголовніша інтеграційна точка:

```text
POST /api/solve -> читаємо дані з БД -> викликаємо solver -> отримуємо результати -> записуємо назад в БД -> віддаємо фронтенду
```

Тут ця людина повинна:

- взяти вхідні таблиці
- передати їх у форматі, який очікує solver
- прийняти output Тимура
- зберегти:
  - `routes`
  - `route_cargo`
  - зміни в `warehouse_stock.reserved_kg`

#### 5. Підтримати логіку складу

Коли користувач на складі натискає `Відвантажено`, саме backend повинен:

- прийняти дію
- зменшити `quantity_kg`
- зменшити `reserved_kg`
- за потреби оновити пов'язані записи

Тобто ця людина відповідає за **правильність стану системи після відвантаження**.

#### 6. Підготувати конфіг середовища

Мінімально:

- локальний запуск сервера
- `.env`
- CORS для фронтенду
- коректні шляхи до SQLite

Це не окрема велика “девопс-роль”, але це обов'язковий технічний фундамент.

### Як виглядає її результат

Результат роботи цієї людини:

- сервер запускається локально
- база створюється
- `demo_data` імпортується
- API повертає живі дані
- `POST /api/solve` уміє працювати зі solver-ом Тимура
- `POST /api/stock/update` оновлює складські залишки

Тобто після її роботи продукт уже має:

```text
дані + логіку + API
```

### Що в цю роль не входить

Щоб не було плутанини, ця людина **не повинна**:

- робити дизайн сайту
- малювати карту
- збирати React-інтерфейс
- придумувати алгоритм маршрутизації за Тимура

## 4. Роль 2 — Frontend / SPA / UI

### Суть ролі

Ця людина будує **веб-інтерфейс продукту**.

Вона відповідає за те, щоб менеджер і працівник складу реально могли користуватись системою.

Простою мовою:

```text
ця людина перетворює дані з API в зрозумілий, красивий і практичний сайт
```

### Чому ця роль критична

Без неї:

- буде API, але ним не зможуть нормально користуватись
- буде solver, але його результат не буде видно
- не буде демонстрації продукту для суддів

Саме frontend робить із логіки **продукт, який можна показати і в якому можна працювати**.

### Що саме вона повинна зробити

#### 1. Побудувати SPA сайт

Потрібен один веб-додаток з двома основними зонами:

- `Диспетчер`
- `Склад`

Найкращий формат для MVP:

```text
один SPA -> перемикач ролей / вкладки
```

#### 2. Зробити екран диспетчера

Це головний desktop-інтерфейс.

Що там має бути:

- карта мережі
- вузли:
  - заводи
  - склади
  - магазини
- маршрути машин
- таблиця поточного попиту
- таблиця маршрутів
- кнопка `Solve`
- кнопка `Urgent`

Менеджер повинен бачити:

- де критичні точки
- які машини куди їдуть
- що система порахувала

#### 3. Зробити екран складу

Це спрощений mobile-friendly інтерфейс.

Що там має бути:

- список машин на завантаження
- cargo для конкретної машини
- куди їде товар
- який товар
- скільки кг
- кнопка `Відвантажено`
- блок поточних залишків складу

Це дуже важливо, бо складський сценарій у концепті не декоративний — це один із ключових user flow продукту.

#### 4. Підключити API бекенду

Frontend повинен брати дані тільки через API.

Ключові запити:

- `GET /api/network`
- `GET /api/routes`
- `GET /api/demand`
- `GET /api/stock`
- `POST /api/solve`
- `POST /api/stock/update`
- можливо `POST /api/urgent`

#### 5. Правильно показати результат solver-а Тимура

Тут дуже важливий момент.

Frontend не просто “малює дані”, а повинен **правильно інтерпретувати output solver-а**:

- який маршрут належить якій машині
- які stop-и в якій послідовності
- який cargo прив'язаний до маршруту
- які маршрути активні
- які магазини пріоритетні

Тобто frontend має бути максимально узгоджений із тим, що повертає backend після роботи solver-а.

### Як виглядає її результат

Результат роботи цієї людини:

- сайт запускається
- є режим `Диспетчер`
- є режим `Склад`
- менеджер може натиснути `Solve`
- менеджер бачить карту, попит і маршрути
- склад бачить, що вантажити
- склад може натиснути `Відвантажено`

Тобто після її роботи продукт має:

```text
інтерфейс + UX + демонстраційність + реальний сценарій використання
```

### Що в цю роль не входить

Щоб не було плутанини, ця людина **не повинна**:

- будувати базу даних
- писати логіку solver-а
- зберігати напряму дані в SQLite
- вигадувати бізнес-логіку за backend

---

## 5. Як ці дві ролі взаємодіють з Тимуром

Це ключова частина.

### Тимур -> Backend

Тимур дає:

- формат вхідних даних для solver-а
- формат вихідних даних
- структуру результату маршрутизації

Backend повинен:

- передати solver-у правильний input
- прийняти його output
- зберегти в БД
- віддати через API

### Backend -> Frontend

Backend дає:

- стабільні endpoint-и
- JSON формати
- актуальні маршрути
- попит
- складські залишки

Frontend повинен:

- акуратно прочитати ці дані
- відобразити їх у правильних блоках UI
- відправляти назад дії користувача

### Тимур -> Frontend

Напряму вони майже не повинні бути сильно зв'язані.

Взаємодія йде через backend.

Але frontend повинен розуміти сенс результату Тимура:

- чому є `priority`
- що таке `route_cargo`
- чому машина везе саме ці точки
- що означає `leg`

## 6. Як це має виглядати в готовому продукті

Щоб усім було зрозуміло, ось правильний робочий сценарій:

### Крок 1

Backend уже має:

- БД
- network data
- demand
- stock
- settings

### Крок 2

Менеджер відкриває сайт.

Frontend показує:

- карту
- попит
- наявний стан мережі

### Крок 3

Менеджер натискає `Solve`.

Frontend:

- викликає `POST /api/solve`

Backend:

- читає дані
- викликає solver Тимура
- записує `routes`
- записує `route_cargo`
- резервує товар у `warehouse_stock`

### Крок 4

Frontend оновлює:

- таблицю маршрутів
- карту маршрутів
- список машин для складу

### Крок 5

Працівник складу відкриває вкладку `Склад`.

Frontend показує:

- які машини вантажити
- що саме вантажити
- скільки кг
- на які магазини

### Крок 6

Склад натискає `Відвантажено`.

Frontend:

- відправляє `POST /api/stock/update`

Backend:

- оновлює `quantity_kg`
- оновлює `reserved_kg`

### Крок 7

У системі синхронізується реальний стан складу.

І це вже виглядає як цілісний продукт, а не набір окремих файлів.

## 7. Важливість кожного аспекту

Щоб команда не знецінювала якісь частини, ось чому кожен аспект важливий.

### Аспекти Backend / API людини

#### База даних
Важлива, бо це джерело істини для мережі, попиту, складу, маршрутів.

#### API
Важливе, бо це міст між алгоритмом і сайтом.

#### Інтеграція з solver-ом
Важлива, бо саме тут математика стає частиною продукту.

#### Оновлення stock / routes
Важливе, бо без цього система не відображає реальний стан.

#### Конфіг і середовище
Важливі, бо без них усе буде нестабільно запускатись або не з'єднається між собою.

### Аспекти Frontend / SPA людини

#### Диспетчерський dashboard
Важливий, бо це головний екран прийняття рішень.

#### Складський інтерфейс
Важливий, бо саме він закриває реальну операційну дію “відвантажити товар”.

#### Інтеграція з API
Важлива, бо сайт без живих даних нічого не вартий.

#### Візуалізація маршрутів і попиту
Важлива, бо це головна цінність продукту на демо і для користувача.

#### Mobile-friendly склад
Важливий, бо складом часто користуються з телефону або маленького екрана.

## 9. Короткий підсумок

Правильна структура команди така:

### Тимур
Рахує, **як система будує маршрути**.

### Backend / API людина
Забезпечує, **як ці розрахунки живуть у системі, зберігаються і віддаються назовні**.

### Frontend / SPA людина
Забезпечує, **як люди бачать ці розрахунки і взаємодіють із ними через сайт**.

Якщо дуже коротко:

```text
Тимур = мозок маршрутизації
Backend = системний хребет
Frontend = обличчя і робочий інтерфейс продукту
```
