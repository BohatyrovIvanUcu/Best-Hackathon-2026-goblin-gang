# Frontend / SPA / UI Brief

Цей документ описує роль людини, яка відповідає за **веб-інтерфейс LogiFlow**. Це не роль алгоритміста і не роль людини, яка працює з БД напряму. Це роль, яка перетворює дані системи в **зрозумілий, сильний і демонстраційний продукт** для двох реальних користувачів:

- менеджера / диспетчера
- працівника складу

## 1. Місія ролі

Ця людина відповідає за те, щоб LogiFlow виглядав не як “набір JSON і таблиць у базі”, а як **справжній продукт**, яким можна користуватись.

Саме вона забезпечує весь користувацький шар:

```text
API -> сторінки -> карта -> таблиці -> кнопки дій -> робочий сценарій для менеджера і складу
```

Простою мовою:

```text
ця людина перетворює внутрішню логіку системи в сайт, через який нею реально можна керувати
```

Без цієї ролі:

- буде solver Тимура, але його результат не буде видно й зрозуміло
- буде API, але ним незручно користуватись
- не буде сильного демо для суддів
- не буде практичного інтерфейсу для складу

## 2. Що це за роль на практиці

Це роль типу:

```text
Frontend Developer / SPA Developer / UI Integration Owner
```

Ця людина:

- будує SPA
- робить вкладки або режими ролей
- підключає REST API
- показує мережу, попит, маршрути і cargo
- робить кнопки `Solve`, `Urgent`, `Відвантажено`
- забезпечує адекватний UX для desktop і mobile

Вона не займається:

- написанням solver-а
- прямою роботою з SQLite
- серверною бізнес-логікою
- збереженням даних поза API

## 3. Як ця роль вписується в концепт LogiFlow

У концепті продукту є дві реальні робочі ролі:

- диспетчер
- склад

А сам продукт повинен показати три ключові речі:

1. мережу доставки
2. рішення системи по маршрутах
3. операційну дію по відвантаженню

Frontend / SPA людина потрібна саме для того, щоб це побачили користувач і судді.

Вона перетворює:

- вузли мережі -> в карту і візуальну модель
- попит -> у зрозумілу таблицю пріоритетів
- результат solver-а -> в маршрути і список машин
- `route_cargo` -> у конкретне завдання для складу

## 4. Що саме ця людина повинна зробити

### 4.1. Побудувати SPA сайт

Рекомендований стек уже описаний у [TASKS.md](/Users/yurleomel/Desktop/BestHackaton/TASKS.md):

- `React 18`
- `TypeScript`
- `Vite`
- `Tailwind`

Мінімальний результат:

- сайт локально запускається
- є один SPA
- є перемикач між двома робочими режимами

Найкращий формат для MVP:

```text
один додаток -> дві вкладки або два режими: "Диспетчер" і "Склад"
```

### 4.2. Зробити екран диспетчера

Це головний desktop-інтерфейс продукту.

Диспетчер має бачити:

- карту мережі
- де заводи, де склади, де магазини
- які магазини `CRITICAL`, `ELEVATED`, `NORMAL`
- які машини куди їдуть
- результат після `Solve`

Що обов'язково має бути на цьому екрані:

- кнопка `Solve`
- карта мережі
- таблиця маршрутів
- таблиця попиту
- можливість викликати `Urgent`

Це головний “розумний” екран продукту, який показує цінність системи.

### 4.3. Зробити екран складу

Це спрощений mobile-friendly інтерфейс.

Його логіка не про аналітику, а про конкретну операцію:

```text
подивився, що вантажити -> завантажив -> підтвердив відвантаження
```

Що обов'язково має бути:

- список машин, які треба готувати
- cargo по конкретній машині
- куди саме їде товар
- який товар
- скільки кг
- кнопка `Відвантажено`
- поточні залишки складу

Цей екран має бути дуже простим, великим і зручним навіть на телефоні.

### 4.4. Підключити API бекенду

Frontend бере всі дані тільки з API. Жодної прямої роботи з базою бути не повинно.

Ключові endpoint-и:

- `GET /api/network`
- `GET /api/routes`
- `GET /api/demand`
- `GET /api/stock`
- `POST /api/solve`
- `POST /api/stock/update`

Можливо також:

- `POST /api/urgent`

Frontend повинен:

- правильно читати відповіді
- показувати стани завантаження
- реагувати на помилки
- оновлювати дані після дій користувача

### 4.5. Правильно показати результат solver-а Тимура

Тут важливий нюанс: frontend не просто показує “якийсь маршрут”, а повинен коректно інтерпретувати результат алгоритму.

Потрібно розуміти:

- що таке `leg`
- що таке `stops`
- що таке `route_cargo`
- чому певна машина має саме цей маршрут
- чому певний магазин має саме такий priority

Тобто frontend повинен поважати сенс output-а, який дав solver через backend.

### 4.6. Дати сильний продуктовий UX

Ця роль важлива не тільки технічно, а й презентаційно.

Інтерфейс має:

- одразу показувати, що це логістичний продукт
- давати менеджеру відчуття контролю
- давати складу максимально простий сценарій
- робити цінність solver-а видимою

Тобто тут важлива не лише функціональність, а й те, як вона подана.

### 4.7. Тримати frontend дисциплінованим і безпечним

Frontend у цьому проєкті не повинен:

- містити секрети або ключі доступу
- дублювати серверну бізнес-логіку
- напряму працювати з БД
- “вигадувати” дані, яких немає в API

Що важливо:

- усі дані приходять тільки з backend API
- усі зміни відбуваються тільки через backend
- frontend зберігає тільки UI-state
- критична логіка стану системи залишається на сервері

Простою мовою:

```text
ця людина відповідає за чистий, зрозумілий і безпечний клієнтський шар, який не ламає архітектуру продукту
```

## 5. Що ця людина отримує від backend і Тимура

### Від backend

Frontend отримує:

- готове API
- network data
- попит
- routes
- складські залишки
- дії `Solve`, `Urgent`, `Відвантажено`

### Від Тимура

Напряму frontend не підключає solver.

Але frontend повинен розуміти бізнес-сенс результату Тимура:

- порядок stop-ів у маршруті
- пріоритети магазинів
- прив'язку cargo до маршруту
- метрики маршруту:
  - кілометраж
  - вартість
  - час

Тобто frontend не пише алгоритм, але **правильно показує його результат**.

## 6. Що ця людина віддає системі в результаті

Результат роботи цієї людини:

- є SPA сайт
- є зрозумілий dispatcher dashboard
- є mobile-friendly складський режим
- менеджер може натиснути `Solve`
- менеджер бачить маршрути і критичні точки
- склад бачить, що вантажити
- склад може натиснути `Відвантажено`

Тобто після роботи цієї ролі продукт має:

```text
видимий, керований і демонстраційний інтерфейс
```

## 7. Важливість кожного аспекту цієї ролі

### Диспетчерський dashboard

Важливий тому, що саме там видно інтелект продукту:

- критичні точки
- рішення solver-а
- мережу доставок
- контроль над логістикою

Це основний екран для демо і для менеджерського сценарію.

### Карта мережі

Важлива тому, що логістичний продукт без візуального шару програє по зрозумілості.

Карта дозволяє одразу побачити:

- географію
- маршрути
- кластери складів
- проблемні магазини

### Таблиця попиту

Важлива тому, що менеджер має бачити не просто точки на карті, а причину, чому система взагалі щось рахує.

### Таблиця маршрутів

Важлива тому, що саме вона формалізує рішення solver-а в операційний результат:

- яка машина
- який маршрут
- яка довжина
- яка вартість
- який час

### Складський інтерфейс

Важливий тому, що це не “додаткова фіча”, а реальна бізнес-операція:

- система не просто рахує
- система доводить результат до дії на складі

### Mobile-friendly підхід

Важливий тому, що складський інтерфейс скоріше за все дивляться з телефону або малого екрана. Якщо він буде незручний, реальний user flow зламається.

### Дисципліна інтеграції з API

Важлива тому, що frontend дуже легко може почати жити “своєю логікою”, і тоді UI та система всередині почнуть розходитись.

У цьому проєкті frontend повинен максимально точно відображати те, що реально повернув backend після роботи solver-а.

## 8. Межі відповідальності

Що входить у цю роль:

- SPA
- layout
- UI для менеджера
- UI для складу
- інтеграція з API
- карта
- таблиці
- картки машин і cargo
- кнопки `Solve`, `Urgent`, `Відвантажено`
- стани loading / error / refresh
- mobile-friendly адаптація

Що не входить:

- база даних
- прямі SQL-операції
- серверні оновлення даних
- бізнес-логіка solver-а
- обрахунок shortest path / assignment / 2-opt

## 9. Як має виглядати успішний результат

Після роботи цієї людини система повинна вміти:

1. відкрити диспетчерський режим
2. показати network на карті
3. показати попит і пріоритети
4. запустити `Solve`
5. оновити маршрути після `Solve`
6. показати список машин і cargo для складу
7. дати натиснути `Відвантажено`
8. оновити UI після змін

Тобто результат цієї ролі:

```text
сайт, через який менеджер і склад реально користуються всією системою
```

## 10. Який має бути ідеальний user flow

Ось як має виглядати правильний досвід користувача:

### Крок 1

Менеджер заходить на вкладку `Диспетчер`.

Бачить:

- карту мережі
- критичні магазини
- поточний попит

### Крок 2

Менеджер натискає `Solve`.

Frontend:

- показує loading
- чекає відповідь від `POST /api/solve`
- після успіху оновлює routes, demand і складські дані

### Крок 3

Менеджер бачить:

- які машини куди їдуть
- як змінилась ситуація
- які точки закриваються

### Крок 4

Працівник складу відкриває вкладку `Склад`.

Бачить:

- машину
- список товарів
- вагу
- магазин призначення

### Крок 5

Після реального завантаження натискає `Відвантажено`.

Frontend:

- відправляє запит у backend
- оновлює картки і залишки після успішної відповіді

### Крок 6

UI показує вже актуальний стан складу.

І в цей момент система виглядає як завершений продукт.

## 11. Що робити в першу чергу

Найкращий порядок роботи:

1. Підняти SPA каркас

6. Підключити кнопки:
   - `Solve`
   - `Відвантажено`
   - потім `Urgent`

## 12. MVP і що не треба переускладнювати

Для MVP не треба:

- дуже складні анімації
- супердетальний design system
- складний offline-first
- web sockets
- ідеальний diff after re-solve
- багато дрібних polish-фіч

Для MVP треба:

- дуже зрозумілий layout
- жива інтеграція з API
- карта
- таблиці
- чіткі кнопки дій
- хороший mobile-friendly склад

## 13. Дизайн-напрямок

Інтерфейс не повинен бути випадковим або “типовим шаблоном”.

Потрібно тримати такий підхід:

- менеджерський екран має відчуватись як dashboard управління мережею
- складський екран має відчуватись як операційний інструмент
- пріоритети мають бути візуально очевидні
- маршрут і cargo мають читатись швидко
- на складі має бути мінімум шуму і максимум ясності

Для складу особливо важливо:

- великі кнопки
- великі картки
- мінімум дрібного тексту
- хороший mobile layout
