## 1. Місія ролі

Ця людина відповідає за те, щоб продукт **жив як система**, а не існував у вигляді окремих файлів.

Саме вона забезпечує весь ланцюжок:

```text
CSV / demo_data -> SQLite -> API -> виклик solver-а Тимура -> routes / cargo / stock updates -> дані для frontend
```

Простою мовою:

```text
ця людина з'єднує дані, алгоритм і сайт в один робочий продукт
```

Без цієї ролі:

- Тимур матиме алгоритм, але не буде нормального способу його викликати
- frontend матиме сайт, але без живих даних
- складський сценарій і кнопка `Відвантажено` не матимуть реального ефекту

## 2. Що це за роль на практиці

Це роль типу:

```text
Backend Engineer / API Engineer / Database & Integration Owner
```

Ця людина:

- піднімає локальний сервер
- створює і заповнює базу даних
- описує і реалізує API
- приймає дії від frontend
- викликає solver Тимура
- зберігає і повертає результати маршрутизації

Вона не займається:

- дизайном сайту
- картою
- React-компонентами
- математикою маршруту замість Тимура

## 3. Як ця роль вписується в концепт LogiFlow

У концепті продукт має таку логіку:

```text
Завод -> Склад -> Магазин
```

І головна цінність системи:

- отримати дані мережі
- проаналізувати попит
- порахувати маршрути
- дати менеджеру інструмент керування
- дати складу інструмент відвантаження

Backend / API людина потрібна саме для того, щоб цей потік реально працював:

1. завантажити й зберегти дані мережі
2. дати solver-у Тимура нормальний input
3. отримати від solver-а routes і cargo
4. записати це в БД
5. віддати це frontend-у
6. прийняти назад дії складу і змінити реальний стан запасів

Рекомендований стек уже зафіксований у [TASKS.md](/Users/yurleomel/Desktop/BestHackaton/TASKS.md):

### 4.2. Реалізувати схему БД

Треба створити SQLite schema за [DATABASE.md](/Users/yurleomel/Desktop/BestHackaton/DATABASE.md).

Обов'язкові таблиці:

- `nodes`
- `edges`

- `trucks`
- `warehouse_stock`
- `demand`
- `routes`
- `route_cargo`

Чому це критично:

- на ці таблиці опирається і solver, і frontend
- без правильної схеми далі все почне розходитись по форматах

### 4.3. Зробити імпорт demo_data

Потрібно завантажувати CSV з `demo_data/` в SQLite.

У цьому шматку важливо:

- зчитати всі CSV
- провалідувати колонки
- коректно привести типи
- акуратно заповнити таблиці
- мати можливість повторного імпорту без хаосу

Чому це важливо:

- команда одразу отримує стабільний demo dataset
- frontend може розроблятись паралельно вже на живих даних
- Тимур може тестувати solver не на вигаданому JSON, а на реальній схемі продукту

### 4.4. Реалізувати REST API

Ця частина повинна бути максимально прив'язана до [API_CONTRACT.md](/Users/yurleomel/Desktop/BestHackaton/API_CONTRACT.md).

Мінімально потрібні endpoint-и:

- `GET /api/network`
- `GET /api/routes`
- `GET /api/demand`
- `GET /api/stock`

Бажано також:

Ключовий принцип:

```text
Frontend не повинен вигадувати структуру відповідей. Backend має віддавати стабільний і передбачуваний контракт.
```

### 4.5. Інтегрувати solver Тимура

Це центральна задача цієї ролі.

Після того як Тимур віддає готовий solver, backend повинен підключити його в один чіткий сценарій:

```text
POST /api/solve
-> читаємо nodes, edges, trucks, warehouse_stock, demand, settings
-> формуємо input для solver-а
-> запускаємо solver
-> отримуємо routes + route_cargo + route metrics
-> записуємо результат у БД
-> оновлюємо reserved_kg
-> повертаємо фронтенду готовий результат
```

Від цієї людини очікується, що вона:

- не переписує алгоритм Тимура
- не ламає його формат
- акуратно адаптує input/output до БД та API

### 4.6. Підтримати складський workflow

Концепт LogiFlow не закінчується на побудові маршрутів. Важлива реальна операція:

```text
Склад побачив, що завантажити -> відвантажив -> система оновила залишки
```

Саме backend відповідає за те, щоб після кнопки `Відвантажено`:

- зменшився `quantity_kg`
- зменшився `reserved_kg`
- стан системи залишився консистентним

Це одна з найважливіших бізнес-дій у всьому продукті.

### 4.7. Підготувати середовище і конфіг

Мінімум, який має бути:

- `.env` для конфігів
- шлях до SQLite
- CORS для локального frontend
- зрозумілий локальний запуск
- дефолтні settings

Тут не треба йти в overengineering, але базовий порядок має бути.

### 4.8. Забезпечити базову безпеку і дисципліну даних

Для хакатонного MVP тут не потрібна велика enterprise-безпека, але потрібна нормальна технічна гігієна.

Що має забезпечити ця людина:

- не зберігати секрети прямо в коді
- винести конфіги в `.env`
- не давати frontend-у прямого доступу до БД
- приймати всі зміни стану тільки через API
- провалідовувати payload-и на вході
- не ламати цілісність `routes`, `route_cargo`, `warehouse_stock`

Простою мовою:

```text
ця людина відповідає не лише за те, щоб усе працювало, а й за те, щоб система не була хаотичною і небезпечною
```

## 5. Що ця людина отримує від Тимура

Від Тимура ця роль отримує не сайт і не базу, а **solver як окремий модуль**.

Логічно очікувати, що Тимур передасть:

- функцію або модуль `solve(...)`
- зрозумілий input format
- зрозумілий output format
- структуру маршруту
- структуру cargo
- метрики маршруту

Backend людина повинна бути готова прийняти щось на кшталт:

```python
result = solve(
    nodes=...,
    edges=...,
    trucks=...,
    warehouse_stock=...,
    demand=...,
    settings=...,
)
```

І на виході отримати щось на кшталт:

```python
{
    "routes": [...],
    "route_cargo": [...],
    "unassigned": [...],
}
```

Навіть якщо точний формат ще уточнюється, ця роль повинна тримати в голові саме таку інтеграційну модель.

## 6. Що ця людина віддає фронтенду

Frontend не має працювати з SQLite напряму. Усе тільки через API.

Тому результат роботи backend людини для frontend команди:

- стабільний `GET /api/network`
- стабільний `GET /api/routes`
- стабільний `GET /api/demand`
- стабільний `GET /api/stock`
- робочий `POST /api/solve`
- робочий `POST /api/stock/update`

Це означає, що frontend може:

- намалювати карту
- показати пріоритети магазинів
- показати маршрути машин
- показати список завантаження на склад
- оновлювати стан після дій користувача

## 7. Важливість кожного аспекту цієї ролі

### База даних

Важлива тому, що це єдина точка правди для:

- мережі
- машин
- попиту
- запасів
- маршрутів

Якщо схема буде крива або нестабільна, далі зламається і solver, і frontend.

### API

Важливе тому, що це міст між системою всередині і людьми, які користуються продуктом через сайт.

Без API немає інтеграції.

### Інтеграція зі solver-ом

Важлива тому, що саме тут математика стає бізнес-функцією продукту.

До цього є лише код. Після цього є вже реальна дія: `Solve`.

### Оновлення складських залишків

Важливе тому, що без цього складська частина продукту фальшива. Кнопка `Відвантажено` повинна реально змінювати дані.

### Конфіг і стабільний запуск

Важливі тому, що команда має швидко піднімати середовище локально і не витрачати час на хаос із портами, шляхами і CORS.

### Базова безпека і цілісність даних

Важливі тому, що продукт працює з реальним станом складу, маршрутів і запасів.

Якщо неправильно оновити `reserved_kg`, `quantity_kg` або маршрути, то сайт покаже картинку, яка не відповідає реальності системи.

## 11. MVP і що не треба переускладнювати

Для MVP не треба:

- складна авторизація
- велика безпека enterprise-рівня
- production-ready observability
- складний деплой pipeline
- websocket-и
- ідеальний offline sync

Для MVP треба:

- стабільний локальний сервер
- правильна БД
- робочий API
- інтеграція solver-а
- коректна зміна стану системи
