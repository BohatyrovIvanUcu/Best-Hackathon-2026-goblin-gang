# LogiFlow — Roadmap (Перспективи розширення)

## Рівень 3: Автоматичне оновлення попиту через Webhook / API

### Ідея

Система клієнта (касова система, WMS, ERP) автоматично надсилає нам зміни попиту в реальному часі. Нова продажа в магазині → попит оновився → система перераховує пріоритети без участі людини.

### Як це працює

```
Каса Аврори → POST /api/demand/update → LogiFlow backend
                                              ↓
                                    перерахунок пріоритетів
                                              ↓
                             якщо точка стала CRITICAL → auto re-solve
                                              ↓
                                диспетчер отримує сповіщення
```

### API endpoint

```http
POST /api/demand/update
Authorization: Bearer {api_key}
Content-Type: application/json

{
  "node_id": "STORE_7",
  "resource_id": "product_A",
  "current_stock": 12,
  "min_stock": 100
}
```

Система сама розраховує пріоритет і вирішує чи потрібен re-solve.

### Типи інтеграцій

| Система клієнта | Спосіб підключення |
|---|---|
| 1С, SAP | Webhook на кожну зміну залишків |
| Excel / Google Sheets | Scheduled sync раз на годину |
| Власна WMS | REST API двосторонній обмін |
| Касова система | Event-driven: кожен продаж → update |

### Що це дає клієнту

- Нульова участь людини в оновленні даних
- Реакція на зміну попиту за секунди, не за годину
- Повна інтеграція в існуючу інфраструктуру клієнта

## Окремі додатки під кожну роль

### Водійська програма (Mobile)
- Timeline маршруту з часами прибуття
- Поточна зупинка на першому місці — назва, адреса, товар
- Кнопка "Доставлено" після кожної зупинки
- Urgent одним тапом
- Повний offline режим (PWA + IndexedDB)

### Магазинна програма (Mobile/Desktop)
- Поточний стан запасів з індикатором критичності
- Форма оновлення попиту
- Urgent Request: товар → кількість → надіслати
- Статус: коли очікується наступна доставка і яка вантажівка їде

### Заводська програма (Desktop)
- Список Leg1 замовлень: що відвантажити на склади
- Підтвердження відвантаження партії
- Поточні виробничі залишки

### Повноцінна авторизація (JWT + RBAC)
Після MVP — повноцінний логін з ролями dispatcher / warehouse / driver / store_manager / factory_manager.

**JWT:** токен з роллю і exp (8 год), refresh token в httpOnly cookie (не localStorage → захист від XSS).

**RBAC таблиця прав:**
| Дія | dispatcher | warehouse | driver |
|---|:---:|:---:|:---:|
| Завантажити CSV | ✅ | ❌ | ❌ |
| Solve | ✅ | ❌ | ❌ |
| Urgent | ✅ | ✅ | ✅ |
| Всі маршрути | ✅ | ❌ | ❌ |
| Свій маршрут | ✅ | — | ✅ |
| Підтвердити відвантаження | ❌ | ✅ | ❌ |
| Audit log | ✅ | ❌ | ❌ |

**Audit log:** незмінна таблиця — хто, що, коли зробив. Insert only.

**bcrypt** для паролів (cost factor 12).

**Rate limiting:** slowapi на `/api/solve` — 10 запитів/хв/користувач.

## Інші перспективи розширення

### ML-прогнозування попиту

Замість того щоб реагувати на дефіцит — передбачати його заздалегідь на основі:
- Сезонності (свята, акції)
- Історії продажів
- Погодних умов (для певних товарів)

```
Поточна логіка:  дефіцит виник → реагуємо
ML логіка:       дефіцит виникне через 2 дні → готуємось зараз
```

### Режим водія EC 561/2006 (повний timeline)

Складне обмеження яке не варто вбудовувати в ядро оптимізатора на MVP — занадто багато граничних випадків.

**Що включає повна реалізація:**

Два типи рейсів:
| Тип | Умова | Логіка |
|---|---|---|
| Local | ≤ 9 год їзди | Один день, перерви всередині |
| Long-haul | 9–18 год | Дводенний, ночівля в rest_stop |
| Multi-day | > 18 год | Попередження: 2 водії або 3+ дні |

Ліміти EC 561/2006 (діє в Україні з 26.07.2026):
- Макс. водіння: 9 год/день (до 10 год — 2 рази/тиждень)
- Перерва: 45 хв після кожних 4.5 год
- Відпочинок: мінімум 11 год між змінами

Повний timeline з автоматичними перервами:
```
08:00  ▶ Виїзд
09:45  📍 Склад Північ
11:30  ☕ Перерва 45 хв (EC 561/2006)
12:15  📍 Аврора Позняки 🔴
14:40  📍 Аврора Троєщина
15:50  ▶ Повернення
```

Точки відпочинку (rest_stop) — новий тип вузла в nodes.csv для long-haul ночівлі. Пріоритет: власний склад клієнта → партнерська стоянка.

### Два водії (team driving)

Для маршрутів > 18 год — планування з двома водіями в одній машині. Поки один їде — інший відпочиває в кабіні. Безперервний рух без ночівлі.

### Мультитенантність

Одна інсталяція LogiFlow обслуговує кількох клієнтів одночасно (Аврора, Сільпо, АТБ) з повною ізоляцією даних між ними.

### Інтеграція з тахографом

Отримувати реальні дані про час їзди і відпочинку водія напряму з цифрового тахографа (G2V2 — обов'язковий в Україні з 30.06.2026). Автоматичний контроль дотримання EC 561/2006.

### Мобільний додаток для водія (native)

Окремий iOS/Android додаток з:
- Навігацією (інтеграція з Google Maps / Waze)
- Підтвердженням доставки з фото
- Цифровим підписом отримувача
- Офлайн-режимом при відсутності зв'язку
